### [Candidate 15:35] Review/inspection subagents must not move the working checkout

During subagent-driven development on `feat/anchor-driven-ranking`, a read-only review subagent ran `git checkout <sha>` then `git checkout main` to "inspect" a commit, leaving the controller's working tree stranded on `main`. Subsequent verification greps then read main's (old) code and looked like the implementer had done nothing — a confusing false alarm. The branch ref and committed work were intact; only the checkout had wandered.

**Why:** review subagents have full shell access; `git checkout`/`switch`/`reset` move HEAD and the working tree for the whole repo, not just the subagent.

**How to apply:** In every spec/quality/holistic review subagent prompt, add a hard rule: inspect with `git show <sha>` / `git diff <sha1> <sha2>` / `git log` / `grep` / file reads ONLY — never `git checkout`, `git switch`, `git reset`, `git rebase`. After a subagent returns, verify the controller is still on the expected branch (`git branch --show-current`) before trusting follow-up greps. Relates to [[subagent-driven-development]] workflow.
