### [Candidate 10:55] fully asynchronous per-action gh fetches implemented (issue #11)

- **Root-cause analysis & design:** The startup gist list and local scans were already off-thread, but all other `gh` actions (diff, download, upload, create, delete, description edit, preview) were blocking the main thread. This froze the UI, preventing redraws (e.g. on resize) and making the app feel unresponsive.
- **Async architecture:**
  - Added `bg_task_msg: Option<String>` to `AppState`.
  - Defined `BgTaskOutcome` enum in `src/tui.rs` carrying either `Result<String, String>` (for fetches) or `Result<(), String>` (for commands), along with the captured context (paths, labels, keys) needed for main-thread state transitions.
  - Spawns a background thread for every blocking `gh` action. The thread executes the command, maps the output, and sends a `BgTaskOutcome` over a channel.
  - The main loop polls `bg_rx` alongside `gist_rx` and `local_rx`. On outcome, it clears the loading overlay and applies the mutations purely to `AppState`.
- **UI & responsiveness improvements:**
  - The loading overlay is now drawn on top of the active background screen, so the app remains visible instead of blanking out.
  - The event loop is fully responsive to resize events and redraws during background tasks.
  - Added cooperative cancellation: pressing `Esc` while a task is running clears the loading message and drops the receiver, safely ignoring the task's completion.
- **Code cleanup:** Removed all blocking sync functions (`refresh_gists`, `fetch_then`, `draw_loading`, `preview_diff`, `preview_content`, `upload_add`, `upload_preview`, `upload_replace`, `create_gist`, `delete_gist`, `remove_file`, `apply_description`). Async refreshes are now triggered by setting `state.loading = true` and spawning a fresh gist fetch.
- **Verification:** All 154 tests pass; `cargo fmt` and `cargo clippy --all-targets -- -D warnings` pass clean.
